## 发现的问题

### 1. 死代码

- `backend/src/utils/cleanup.js` 未被任何地方引用
- 该文件依赖不存在的 `fileService`，无法正常工作

### 2. 数据库利用率低

- SQLite 只存储剪贴板文本（ContentItem 表）
- 文件和图片信息未入库，无法追踪元信息（上传时间、来源设备等）

### 3. 文件存储设计缺陷

- 文件使用原始文件名存储，同名文件会被覆盖
- 文件和图片分两个目录，但功能相似，导致代码重复
- `config/multer.js` 和 `routes/imageRoutes.js` 各自独立配置 multer

### 4. 缺少基础功能

- 无全局错误处理中间件
- 无请求参数验证
- 文件大小限制不一致（图片限制 5MB，文件无限制）

### 5. 日志系统过于简单

- 只是 `console` 的简单封装
- 无日志级别区分、无文件输出

### 6. 前端问题

- 页面组件直接调用 fetch，无统一的 API 封装层
- 无统一的加载状态管理

## 优化建议

考虑到这是个人家庭局域网使用、部署在树莓派上，建议聚焦于实用性和轻量级。

### 高优先级

| 优化项 | 理由 |
|--------|------|
| 删除 `cleanup.js` 死代码 | 减少混淆，保持代码库整洁 |
| 文件名添加时间戳前缀 | 避免同名文件覆盖 |

### 中优先级

| 优化项 | 理由 |
|--------|------|
| 统一文件/图片的 multer 配置 | 减少重复代码，便于维护 |
| 添加全局错误处理中间件 | 提升应用稳定性和错误响应一致性 |

### 低优先级

| 优化项 | 理由 |
|--------|------|
| 添加环境变量配置文件大小限制 | 提升配置灵活性 |
| 前端封装 API 调用层 | 代码组织更清晰 |

### 不建议实施

- 引入 TypeScript：个人项目收益不大，增加复杂度
- 引入前端状态管理库：当前状态简单，Context API 足够
- 微服务或容器化：过度工程，不符合项目定位
- 文件信息入库：除非需要按时间/设备筛选，否则增加不必要复杂度

## 结论

项目整体架构简洁清晰，符合个人项目的定位。主要问题集中在代码重复和少量死代码，优化后可提升可维护性。无需进行大规模重构。
